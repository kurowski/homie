# Homie â€” Project Brief

## What is this?

**Homie** is a generic Linux environment manager for developers. It handles
both dotfile management and system provisioning in a single tool, driven by
a user-owned git repo of config and dotfiles.

- CLI command: `hm`
- Domain: `homie.sh` (documentation and marketing only â€” not a curl target)
- Tool repo: `github.com/kurowski/homie` (this repo â€” Go source only)
- Go module path: `github.com/kurowski/homie`

---

## Two-repo architecture

There are two distinct repos involved in a Homie setup. They must not be
conflated.

### 1. The Homie tool repo (`github.com/kurowski/homie`)

This repo. Contains only:

- Go source for the `hm` binary
- Documentation website (`docs/website/`)
- `README.md`, `CLAUDE.md`, CI/release config

No dotfiles. No user config. No `homie.toml`. This is a generic tool that any
developer can use.

### 2. A user's environment repo (e.g. `github.com/kurowski/dotfiles`)

Generated by `hm init`. Contains:

- `bootstrap.sh` â€” the curl|bash entrypoint for that user's environment
- `homie.toml` â€” user's config (packages, vars, profile)
- `dotfiles/` â€” files to be symlinked into `$HOME`
- `templates/` â€” files rendered into `$HOME` with var sub and conditionals
- `scripts/` â€” ordered setup steps

This repo is owned and trusted by the user. It references the Homie tool but
is not part of it.

---

## Bootstrap and trust model

A user bootstraps their environment with:

```sh
curl https://raw.githubusercontent.com/kurowski/dotfiles/main/bootstrap.sh | sudo bash
```

The user is curling **their own repo**. They trust themselves, not Homie's
author. This is the correct trust model â€” nobody needs to trust `homie.sh`.

`homie.sh` is a documentation and marketing site only.

The `bootstrap.sh` generated by `hm init` does the following:

1. Detects OS and CPU architecture
2. Downloads the appropriate `hm` binary from
   `github.com/kurowski/homie/releases`
3. Clones the user's environment repo if not already present
4. Runs `hm apply` from the repo root

## The problem Homie solves

Every existing solution forces a tradeoff:

- **chezmoi** manages dotfiles well but uses a copy model with an indirection
  layer â€” you edit files in `~/.local/share/chezmoi`, not where they live.
  Running `chezmoi edit` instead of just editing `~/.zshrc` is daily friction.
- **Ansible** handles provisioning well but is heavy, requires Python, and is
  designed for fleets, not a single developer's environment.
- **GNU Stow** gets the symlink model right but does nothing else.
- **Nix/Home Manager** does everything but is too heavy for ephemeral
  environments like Codespaces and devcontainers.

The common result: people run chezmoi + Ansible + glue scripts as three
separate things. Homie does it all in one, from one repo.

---

## Core design principles

**Symlinks, not copies.** Dotfiles in the user's repo are symlinked into
`$HOME`. Editing `~/.zshrc` edits the repo file directly. Commit and push
with no intermediate step. The checkout *is* the working copy.

**No state.** Every `hm apply` is a full reconciliation pass. Steps are
individually idempotent â€” symlinks are checked before creation, packages
checked before install, templates re-rendered each run. No lockfile, no
state database.

**Config is TOML.** A single `homie.toml` at the user repo root declares
identity, profile, packages, tags, and free-form vars. Matches chezmoi
(the closest analog) and avoids YAML's whitespace/type foot-guns.

**Templating is Go `text/template` + Sprig.** The mainstream choice in
this space â€” chezmoi, Helm, and Argo CD all use it. Users get
conditionals, loops, partials, and ~100 Sprig helpers without Homie
inventing a DSL.

**Friendly output.** Charm.sh TUI stack. Colorful, progress-reported,
pleasant to run.

## homie.toml format

```toml
[user]
name  = "Your Name"
email = "you@example.com"

[profile]
name          = "personal"   # work | personal | devcontainer | ...
default_shell = "zsh"

[packages]
all = ["git", "zsh", "neovim", "tmux", "ripgrep", "fd", "fzf"]
# Optional per-distro overrides, merged onto `all`:
fedora = ["util-linux-user"]
ubuntu = ["fd-find"]

# Tag-keyed sub-tables contribute only when the matching tag is active.
[packages."tag:work"]
fedora = ["kubectl", "helm"]
ubuntu = ["kubectl"]

[packages."tag:personal"]
fedora = ["steam", "tailscale"]

[tags]
extra = ["laptop"]    # merged with auto-detected tags

[vars]
WORK_EMAIL = "you@work.example.com"
EDITOR     = "nvim"
```

`profile.name` becomes an active tag automatically. Auto-detected tags
include `container`, `root`, `amd64`, `arm64`, the detected distro name,
and the short hostname (so `hasTag "coach"` works with no config).
Vars are exposed to templates as `.Vars.NAME` and to `scripts/*.sh` as
environment variables.

Required fields: `user.name`, `user.email`. Everything else is optional.

### Per-host overlay

A repo can ship `hosts/<short-hostname>.toml` files alongside `homie.toml`.
When present, the matching overlay is deep-merged onto the base config
after load: user/profile scalars replace when set, `[packages].*` arrays
and `[tags].extra` append, `[vars]` keys override. This lets the same
checkout serve multiple machines without per-host branches:

```
dotfiles/
  homie.toml              # base, applies everywhere
  hosts/
    coach.toml            # profile=personal + laptop packages
    uceap-dev01.toml      # profile=work + work packages and vars
```

Hostname detection takes the short form (everything before the first dot),
so an FQDN like `coach.lan` matches `hosts/coach.toml`.

## Template format

Templates in `templates/` use `.tmpl` extension and Go's `text/template`
syntax augmented with [Sprig](https://masterminds.github.io/sprig/)
helpers. `templates/.gitconfig.tmpl` renders to `~/.gitconfig`.

```gotmpl
[user]
    name = {{ .Name }}
{{- if hasTag "work" }}
    email = {{ .Vars.WORK_EMAIL }}
{{- else }}
    email = {{ .Email }}
{{- end }}
```

Data exposed to templates:
- `.Name`, `.Email`, `.Profile`, `.DefaultShell` â€” from `[user]` / `[profile]`
- `.Distro`, `.Arch`, `.IsContainer`, `.IsRoot` â€” from environment detection
- `.Tags` â€” string slice of all active tags
- `.Vars` â€” `map[string]string` from the `[vars]` table

Custom funcs on top of Sprig:
- `hasTag <name>` â€” true if the tag is active

Rendering rules:
- Standard `{{ ... }}` delimiters (no customization â€” boring matches
  chezmoi/Helm muscle memory)
- Missing fields fail loudly (`missingkey=error`) â€” typos surface immediately
- Source file mode is preserved (executable templates render executable)

## hm commands

```
hm init              # scaffold a new user environment repo (run once, interactively)
hm apply             # full reconciliation: detect â†’ pre-scripts â†’ packages â†’ backends (alphabetical) â†’ link â†’ render â†’ scripts
hm link              # dotfiles only
hm render            # templates only
hm install           # packages only
hm run               # scripts only (default phase=post; --phase=pre|all)
hm status            # show sync state, no changes
hm doctor            # check for broken symlinks, missing deps, common problems
```

`hm apply` and all subcommands except `hm init` expect to be run from the
root of a user environment repo, or with `HM_REPO` pointing to one.

`hm init` is interactive â€” prompts for name, email, GitHub username, profile,
generates the repo scaffold, and prints next steps.

### Tag-gated dotfile and template trees

Sibling directories named `dotfiles.tag-<X>[.tag-<Y>...]` and
`templates.tag-<X>[.tag-<Y>...]` are processed only when every named tag
is active. Plain `dotfiles/` and `templates/` always apply.

```
dotfiles/                       # always
  .zshrc
dotfiles.tag-work/              # only when hasTag "work"
  .config/autostart/work.desktop
dotfiles.tag-work.tag-kde/      # AND: only when both tags active
  .config/plasma-workspace/...
templates.tag-personal/         # same convention for templates
  .gitconfig.tmpl
```

If two trees produce the same target path, `hm apply` reports an error
â€” overriding by tag should be expressed in a single template with
`{{ if hasTag ... }}`, not by stacking dotfile trees. Two details
worth knowing:
- `link.Plan` fails fast on a dotfile collision (no symlinks created).
- `render.Apply` records each template collision in `Result.Errors`
  and continues with the rest of the trees.

Either way `hm apply` exits non-zero and surfaces the error. `hm doctor`
also lists tag-gated trees that aren't active on the current host as
informational findings.

Tag names appearing in directory suffixes can't contain `.` (the
separator between segments). `dotfiles.tag-fedora.42/` parses as two
segments and rejects on the malformed second one.

### Non-native package backends

Beyond the native package manager (`apt` or `dnf`), `[packages]` accepts
sub-tables for non-native managers. v1 supports `flatpak` and `brew`:

```toml
[packages.flatpak]
all = ["md.obsidian.Obsidian"]
fedora = ["org.localsend.localsend_app"]

[packages.brew]
all = ["fd", "ripgrep", "bat"]

[packages."tag:work".flatpak]
all = ["us.zoom.Zoom"]
```

Each backend mirrors the base shape: `all`, distro keys, and tag-keyed
sub-tables (`[packages."tag:X".flatpak]`). Resolution rules and dedup
are identical to native packages.

Backends are **opt-in by tool presence**. If the backend's CLI tool
isn't on PATH, `hm apply` logs a warning and skips that phase â€” it
doesn't fail. Setting up the flatpak remote (`flatpak remote-add ...`)
or installing brew lives in `scripts/pre-*.sh` so it runs before the
backend's install step.

The Flatpak backend assumes packages come from the `flathub` remote.
References from `flathub-beta`, GNOME nightly, or custom remotes are
not supported by `[packages.flatpak]` â€” add them via `scripts/*.sh`.

Unknown backend names in `[packages]` (e.g. a typo, or `cargo` before
it ships) decode into `Packages.Backends` with a warning rather than
hard-failing the load. `hm apply` and `hm doctor` report them at apply
time, keeping the file forward-compatible with newer hm binaries.

Adding a new backend (cargo, npm, pip, ...) means:
1. Add the name to `config.KnownBackends` (silences the "looks like a
   backend but isn't recognized" warning).
2. Add a `Manager` implementation to `internal/packages` and a case
   in `packages.ForBackend`.

The apply lifecycle iterates whatever backends the user declared
(alphabetical order), so no third registry to update.

### Pre-package scripts (`scripts/pre-*.sh`)

Scripts under `scripts/` whose basename starts with `pre-` run **before**
the package install phase. This is for adding third-party package sources
that `[packages]` would otherwise fail to resolve: dnf COPRs, custom
`.repo` files + GPG keys (VS Code, 1Password, Docker, HashiCorp), apt
`signed-by` keyrings, RPM Fusion release RPMs, Flatpak remote setup, etc.

Every other `scripts/*.sh` runs after as today. Both groups are still
ordered lexically within their phase, and both receive the same env
(`HM_REPO`, `HM_HOME`, `HM_TAGS`, plus `[vars]`). Idempotency remains the
script author's job â€” Homie does not deduplicate.

## Environment detection

The `detect` package determines:

| Field             | Values / notes                                      |
|-------------------|-----------------------------------------------------|
| `Distro`          | `ubuntu`, `debian`, `fedora`, `unknown`             |
| `PackageManager`  | `apt` (Ubuntu/Debian), `dnf` (Fedora), `unknown`    |
| `Arch`            | `amd64`, `arm64`                                    |
| `Hostname`        | short hostname (truncated at first `.`)             |
| `IsContainer`     | bool â€” via `/proc/1/cgroup`, `/.dockerenv`          |
| `IsRoot`          | bool                                                |
| `IsInteractive`   | bool â€” TTY detection                                |

**Supported distros for v1: Ubuntu, Debian, Fedora.**

Arch Linux and other distros should be detected as `unknown` with a clear,
friendly error telling the user it's not yet supported and pointing to
`homie.sh/contributing` to add support. Leave this path clearly marked as
an open contribution opportunity in the code with a `// TODO(contrib)` comment.

## Package manager abstraction

```go
type Manager interface {
    IsAvailable() bool
    IsInstalled(pkg string) bool
    Install(packages []string) error
}
```

**v1 implementations:** `apt` (Ubuntu/Debian), `dnf` (Fedora).

Each checks `IsInstalled` before installing so runs are idempotent without
a state file. Unknown distros get a `NoopManager` that warns and skips.

## TUI behavior

- **Interactive TTY:** full Bubble Tea TUI with spinner, step progress,
  colored output, and a pass/warn/fail summary at the end
- **Non-interactive / CI** (no TTY or `--no-tty`): plain line output, no
  ANSI, suitable for log capture
- `--verbose` available in both modes
- Non-fatal errors collected and surfaced in summary rather than aborting run

## What Homie is not

- Not cross-platform. Linux only for v1.
- Not a secrets manager. Out of scope.
- Not a package version manager. Use `mise`; Homie can call its install script.
- Not for remote machines. Single-machine, single-user.
- Not Nix. Imperative with idempotency guards, not purely declarative.
