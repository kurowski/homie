# Homie — Project Brief

## What is this?

**Homie** is a generic Linux environment manager for developers. It handles
both dotfile management and system provisioning in a single tool, driven by
a user-owned git repo of config and dotfiles.

- CLI command: `hm`
- Domain: `homie.sh` (documentation and marketing only — not a curl target)
- Tool repo: `github.com/kurowski/homie` (this repo — Go source only)
- Go module path: `github.com/kurowski/homie`

## Two-repo architecture

There are two distinct repos involved in a Homie setup. They must not be
conflated.

### 1. The Homie tool repo (`github.com/kurowski/homie`)

This repo. Contains only:

- Go source for the `hm` binary
- Documentation website (`docs/website/`)
- `README.md`, `CLAUDE.md`, CI/release config

No dotfiles. No user config. No `homie.toml`. This is a generic tool that any
developer can use.

### 2. A user's environment repo (e.g. `github.com/kurowski/dotfiles`)

Generated by `hm init`. Contains the user's `homie.toml`, `home/` tree,
optional `hosts/<name>.toml` overlays, optional `scripts/`, and a
`bootstrap.sh` entrypoint. Owned and trusted by the user. The full
layout reference lives at [/docs/quickstart/](https://homie.sh/docs/quickstart/).

---

## Bootstrap and trust model

A user bootstraps their environment with:

```sh
curl https://raw.githubusercontent.com/kurowski/dotfiles/main/bootstrap.sh | sudo bash
```

The user is curling **their own repo**. They trust themselves, not Homie's
author. This is the correct trust model — nobody needs to trust `homie.sh`.

`homie.sh` is a documentation and marketing site only.

The `bootstrap.sh` generated by `hm init` does the following:

1. Detects OS and CPU architecture
2. Downloads the appropriate `hm` binary from
   `github.com/kurowski/homie/releases`
3. Clones the user's environment repo if not already present
4. Runs `hm apply` from the repo root

## The problem Homie solves

Every existing solution forces a tradeoff:

- **chezmoi** manages dotfiles well but uses a copy model with an indirection
  layer — you edit files in `~/.local/share/chezmoi`, not where they live.
  Running `chezmoi edit` instead of just editing `~/.zshrc` is daily friction.
- **Ansible** handles provisioning well but is heavy, requires Python, and is
  designed for fleets, not a single developer's environment.
- **GNU Stow** gets the symlink model right but does nothing else.
- **Nix/Home Manager** does everything but is too heavy for ephemeral
  environments like Codespaces and devcontainers.

The common result: people run chezmoi + Ansible + glue scripts as three
separate things. Homie does it all in one, from one repo.

---

## Core design principles

**Symlinks, not copies.** Dotfiles in the user's repo are symlinked into
`$HOME`. Editing `~/.zshrc` edits the repo file directly. Commit and push
with no intermediate step. The checkout *is* the working copy.

**No state.** Every `hm apply` is a full reconciliation pass. Steps are
individually idempotent — symlinks are checked before creation, packages
checked before install, templates re-rendered each run. No lockfile, no
state database.

**Config is TOML.** A single `homie.toml` at the user repo root declares
identity, profile, packages, tags, and free-form vars. Matches chezmoi
(the closest analog) and avoids YAML's whitespace/type foot-guns.

**Templating is Go `text/template` + Sprig.** The mainstream choice in
this space — chezmoi, Helm, and Argo CD all use it. Users get
conditionals, loops, partials, and ~100 Sprig helpers without Homie
inventing a DSL.

**Friendly output.** Charm.sh TUI stack. Colorful, progress-reported,
pleasant to run.

## Environment detection

The `detect` package determines:

| Field             | Values / notes                                      |
|-------------------|-----------------------------------------------------|
| `Distro`          | `ubuntu`, `debian`, `fedora`, `unknown`             |
| `PackageManager`  | `apt` (Ubuntu/Debian), `dnf` (Fedora), `unknown`    |
| `Arch`            | `amd64`, `arm64`                                    |
| `Hostname`        | short hostname (truncated at first `.`)             |
| `IsContainer`     | bool — via `/proc/1/cgroup`, `/.dockerenv`          |
| `IsRoot`          | bool                                                |
| `IsInteractive`   | bool — TTY detection                                |

**Supported distros for v1: Ubuntu, Debian, Fedora.**

Arch Linux and other distros should be detected as `unknown` with a clear,
friendly error telling the user it's not yet supported and pointing to
`homie.sh/contributing` to add support. Leave this path clearly marked as
an open contribution opportunity in the code with a `// TODO(contrib)` comment.

## Package manager abstraction

```go
type Manager interface {
    Name() string
    IsAvailable() bool
    IsInstalled(pkg string) bool
    Install(packages []string) error
}
```

Native managers (`apt`, `dnf`) are returned by `packages.For(env)`.
Non-native backends (`brew`, `flatpak`, and any reserved future name)
are returned by `packages.ForBackend(name)`. Both share the interface
so the apply phases treat them identically — only the resolution rule
(`PackagesFor` vs `PackagesForBackend`) differs.

Each backend checks `IsInstalled` before installing so runs are
idempotent without a state file. Non-native backends additionally cache
the parsed installed-set with `sync.Once` so a `bucket+Install` round
shells out once per phase rather than 2N times. Unknown distros get a
`NoopManager` that warns and skips.

Adding a backend: extend `config.KnownBackends`, implement `Manager`
in `internal/packages`, add a case to `packages.ForBackend`. The apply
lifecycle iterates whatever backends the user declared (alphabetical),
so no third registry to update.

## The home/ tree (internal model)

`internal/tree` owns the convention. The key entry points:

- `tree.Active(repoDir, base, tags)` — `home/` plus any active
  `home.tag-X[.tag-Y...]/` siblings.
- `tree.Classify(repoDir, base, tags)` — same walk, also returns
  inactive trees (used by `hm doctor`).
- `tree.Resolve(repoDir, home, tags)` — walks every active tree and
  partitions files by `$HOME` target with the **more-specific-wins**
  override rule. Returns `[]Resolved{Source, Target, IsTemplate}`.
  Same-specificity collisions return an error.

Both `link.Plan` and `render.Apply` are thin consumers of
`tree.Resolve`: link takes the non-template entries, render takes the
templates. The override rule is enforced exactly once.

User-facing description of this model: [/docs/dotfiles/](https://homie.sh/docs/dotfiles/).

## TUI behavior

- **Interactive TTY:** full Bubble Tea TUI with spinner, step progress,
  colored output, and a pass/warn/fail summary at the end
- **Non-interactive / CI** (no TTY or `--no-tty`): plain line output, no
  ANSI, suitable for log capture
- `--verbose` available in both modes
- Non-fatal errors collected and surfaced in summary rather than aborting run

## Documentation website

The `docs/website/` directory contains the static site published to `homie.sh`
via GitHub Pages. It should match the style and structure of the author's other
projects (see `github.com/kurowski/potasko` for reference).

Pages on the site:
- `quickstart.md` — install, init, push, bootstrap
- `install.md`, `commands.md`, `config.md`, `dotfiles.md`
- `recipes.md`, `compare.md`, `faq.md`, `contributing.md`

When you change user-visible behavior, update both `hm help <cmd>` and
the relevant Hugo page. The `pages.yml` workflow rebuilds the site on
push to `main`.

Audience note: `contributing.md` is for contributors editing Go code,
so internal package paths and function names are fine there. The rest
of `docs/website/content/` is for end users — keep Go identifiers out
of it (CLI commands, config keys, filesystem conventions are user-facing
and stay).

## Current state

v0.2.0 shipped. The MVP (detect, config, link, render, native packages,
runner, UI, `hm apply` end-to-end, `hm init` scaffold, `bootstrap.sh`
template, `hm status` / `hm doctor`, GitHub Actions release pipeline,
e2e container harness covering Ubuntu/Debian/Fedora, docs site) was
v0.0.2; v0.1.0 added: multi-host overlay, pre-package scripts,
tag-keyed packages, tag-conditional home trees, pluggable backends
(flatpak + brew), unified `home/` tree replacing `dotfiles/` +
`templates/`, more-specific-wins override resolution, docs site
restructure. v0.2.0 added: the `hm home` command, charmbracelet/fang-styled
help and errors, a snap package backend (with `/classic` `/devmode`
`/jailmode` confinement suffixes), and tag-conditional script trees
(`scripts.tag-X/`, parallel to `home.tag-X/`).

**Layout migration** (one-time, for repos created against v0.0.2):
`git mv dotfiles/* home/ && git mv templates/* home/ && rmdir dotfiles
templates`. The `.tmpl` suffix now distinguishes render-from-symlink,
so templates keep their suffix and everything else stays plain. Same
applies to any `dotfiles.tag-X/` / `templates.tag-X/` siblings —
collapse into `home.tag-X/`.

The full curl|bash flow has been verified end-to-end against production
(homie.sh → GitHub releases → a real user dotfiles repo). Next phase is
dogfooding on real environments and filing issues against gaps that
surface.

## What Homie is not

- Not cross-platform. Linux only for v1.
- Not a secrets manager. Out of scope.
- Not a package version manager. Use `mise`; Homie can call its install script.
- Not for remote machines. Single-machine, single-user.
- Not Nix. Imperative with idempotency guards, not purely declarative.
